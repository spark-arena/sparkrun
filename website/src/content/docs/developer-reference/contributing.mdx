## Development setup

```bash
# Clone the repository
git clone https://github.com/spark-arena/sparkrun.git
cd sparkrun

# Install in development mode with dev dependencies
uv pip install -e ".[dev]"
```

## Running tests

```bash
# Full test suite
.venv/bin/python -m pytest tests/ -v

# Single test file
.venv/bin/python -m pytest tests/test_recipe.py -v

# Specific test
.venv/bin/python -m pytest tests/test_cli.py::test_run_command_basic -v

# With coverage
.venv/bin/python -m pytest tests/ --cov=sparkrun --cov-report=term-missing
```

## Linting

sparkrun uses [ruff](https://docs.astral.sh/ruff/) for linting and formatting:

```bash
# Check for lint errors
ruff check src/ tests/

# Auto-format
ruff format src/ tests/
```

Configuration: line-length 140, target Python 3.12.

## Test patterns

All SSH and Docker operations in tests are mocked — no real hosts are needed.

### Key fixtures

| Fixture | Purpose |
|---|---|
| `isolate_stateful` | Autouse fixture that redirects SAF's stateful root to `tmp_path`, preventing tests from touching `~/.config/sparkrun/` |
| `tmp_recipe_dir` | Creates sample v1/v2 recipe files in a temporary directory |
| `cluster_dir` | Temporary cluster configuration directory |
| `hosts_file` | Temporary hosts file |
| `v` | Initialized SAF Variables instance |

### Bootstrap reset

The bootstrap singleton (`_variables`) is reset between tests by the `isolate_stateful` fixture. All core module imports in tests use `sparkrun.core.*` paths:

```python
from sparkrun.core.registry import RegistryManager
from sparkrun.core.recipe import Recipe, find_recipe
```

### Test coverage areas

Tests cover: benchmarking, bootstrap, CLI commands, CLI recipe integration, cluster manager, config, distribution, Docker command generation, GGUF handling, host resolution, InfiniBand, networking, orchestration primitives, recipes, registry (including manifest discovery, fallback merging, shared clones, and reserved name enforcement), runtimes, embedded scripts, SSH execution, kernel tuning, and VRAM estimation.

## Version management

Versions are tracked in `versions.yaml` at the repo root and synced to all package files:

```bash
# Sync versions across packages
python scripts/update-versions.py

# CI-friendly verify (exits non-zero if out of sync)
python scripts/update-versions.py --check
```

This updates version strings in `pyproject.toml` and the `sparkrun-cc-plugin` package.

## Project structure

| Directory | Contents |
|---|---|
| `src/sparkrun/` | Main package source |
| `tests/` | pytest test suite |
| `scripts/` | Build and version management scripts |
| `sparkrun-cc-plugin/` | Claude Code plugin companion package |
| `website/` | Documentation site (Astro/Starlight) |

## PR guidelines

- Run the full test suite before submitting
- Ensure `ruff check` and `ruff format --check` pass
- Run `python scripts/update-versions.py --check` if version-related files changed
- Keep commits focused — one logical change per commit
- Include test coverage for new functionality
